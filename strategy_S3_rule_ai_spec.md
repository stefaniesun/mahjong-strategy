# S3 需求规格:规则基线 AI(陪练 + 数据工厂 + 基准线)

> 策略 AI 项目第三阶段。S1 引擎、S2 状态层(s2.v4)均已定稿验证通过。
> 本阶段做一个**纯规则(无机器学习)的打牌策略**,作用:① 第一个会打牌的对手 ② 后续 S4/S5 的基准线 ③ 自对局产出 S4 监督学习数据(零外部数据来源)④ 检验 S2 状态表示是否够用。
> 交给 AI 编码助手执行。Python 3.10+,依赖少,有单测。

---

## 设计定位(给执行者的关键认知)

**S3 的目标是"会打牌、不打蠢牌",不是"打得强"。**
- 真正的高手水平由 S5 强化学习练出,**不要在 S3 里堆复杂战术**(防守预判、弃和搏大、形势判断等高级决策一律不写——那些留给 S5 自己学,写进 S3 反而会锁死 AI 上限)。
- S3 只实现"无脑正确、不需要复杂权衡"的基础规则,简单稳健、快速可用。
- 衡量标准:4 个 S3 互打,流局率明显低于随机/贪心策略,出牌行为让老玩家看着"没毛病、不蠢"即可。

**S3 必须复用 S1 引擎 + S2 状态层**(import,不重写);它消费 S2 的状态/合法动作,输出动作。这也顺带检验 S2 编码的信息够不够支撑一个像样策略。

## 一、S3 要实现的基础规则(我主导,稳健优先)

### 1. 向听数计算(S3 技术核心)
- 实现"向听数"计算:当前手牌距离听牌还差几步(0=听牌)。同时支持标准型(4面子+1将)和七对型,取两者较小值
- 提供:`shanten(hand)` 返回向听数;`best_discards(hand)` 返回"打掉哪些牌后向听数最优"
- 必须计入定缺约束:有缺门牌时,缺门牌不计入有效进张
- 这是所有出牌决策的基础,要正确高效

### 2. 出牌决策(打牌阶段)
按优先级(稳健启发式):
1. **若手里还有缺门牌 → 优先打缺门牌**(四川麻将铁律,缺门没打完啥都做不了)
2. 缺门打完后 → **选"打掉后向听数最优"的牌**(朝听牌推进)
3. 多个等价选择时 → 优先打孤张(无搭子的牌)、保留搭子和对子
4. **做牌方向倾向"留最多的那门"**:换牌/出牌时,倾向保留数量最多的花色(为清一色等大牌留可能),但不强求——以向听数为主、做牌方向为辅
> 不实现:防守预判、安全牌计算、弃和、形势判断(留给 S5)

### 3. 碰/杠/胡决策
- **能胡就胡**(基础版:可胡则胡,不做"过庄养大牌"的复杂权衡;过庄的合法性由 S2 共享合法性判定 `legality.py` 生成的 legal_mask 保证,S3 只在合法时选胡)
- **碰**:若碰了能让向听数下降(加速听牌)则碰;否则不碰(碰会暴露信息、打乱手牌)。简单规则:碰后向听数严格下降才碰
- **杠**:有暗杠/补杠机会时,若杠不破坏手牌结构(杠的牌本就是用不上的刻子)则杠(吃杠钱+根);否则不杠。基础保守即可
- **抢杠胡**:能抢则抢(合法动作里有就选)

### 4. 开局决策(换三张 + 定缺)
- **定缺**:选当前手牌中**数量最少的那门花色**为缺门(最少的门最容易打光)。若两门数量相同,选更"散"(搭子少)的那门
- **换三张**:从**准备定缺的那门**里选 3 张换出(把要缺的提前清掉);若缺门那门不足3张,补换其他最孤立的同花色3张
- **增量优化:换三张避免资敌**:仍优先从准备定缺的那门换出,但选具体 3 张时优先换孤张,尽量避免把对子、刻子、紧密连张整组送出;在满足"同花色 3 张"硬约束下,挑对自己价值最低、对对手帮助最小的组合。
> 注:换三张的高级技巧(防守性换牌、判断对手)不实现,留给 S5

### 5. S3 增量优化边界(只补基础合理性)

本轮只做两处低成本基础合理性改进,不改变 S3 "新手陪练 + 数据工厂"定位,不加入做大牌意向、动态牌型规划、防守预判等高级战术。

1. **换三张避免资敌**
   - 优先换出孤张、无搭子的同花色牌。
   - 尽量避免换出对子、刻子,也避免把紧密连张整组送给对手。
   - 保持原规则:仍优先从准备定缺的那门换出;缺门不足 3 张时,再从其他门按同样原则选择合法同花色 3 张。
   - 验收:构造含明显对子/刻子的手牌,验证换三张不会优先把对子/刻子整组换出,换出的应以孤张为主。
2. **出牌同向听时按进张数次级判断**
   - 第一优先级不变:先打缺门牌,再选打出后向听数最优的牌。
   - 新增次级判断:多个出牌选择的向听数相同时,选择打出后剩余手牌有效进张数最多的出牌。
   - 进张数按"能让向听数下降的牌"估算,优先使用 S2 `tile_counting` 的已现牌计数扣减剩余张数;缺少已现牌信息时按理论剩余张数估算。
   - 只作为同向听 tie-breaker,不引入复杂牌型价值评估或大牌倾向。
   - 验收:构造两个出牌选择向听数相同但进张数不同的场景,验证 S3 选择进张数更多的那个;不破坏打缺门/降向听主逻辑。

## 三、任务清单

### 任务 1:向听数适配 `shanten.py`(核心,先做)【v4 调整:复用,不重写】
- **向听数只允许一份实现,源头是 S2 的 `hand_analysis.py`**(标准型 + 七对型取最小、计入定缺约束、有效进张,S2 已实现)。S3 不得重写,避免两份实现日后算得不一致、出了分歧不知道信谁
- 本任务做三件事:
  1. 按 S3 需要的接口做薄封装:`shanten(hand)`、`best_discards(hand)`(打掉哪些牌后向听数最优)、`useful_tiles(hand)`
  2. 若 S2 现有能力不满足(如 `best_discards` 缺失),**在 S2 `hand_analysis` 中扩展**,而不是在 S3 里另写一份
  3. 性能确认:向听数会被自对局海量调用,若太慢在 S2 侧优化(优化后 S2 相关单测必须仍全过)
- **验收**:对一批构造手牌(已听牌=0向听、差一张=1向听、七对各阶段、含缺门)计算正确;pytest 用例 ≥30 个手工可验算;断言 S3 封装结果与 S2 `hand_analysis` 直接调用逐一一致

### 任务 2:启发式规则 `heuristics.py` + 策略主体 `rule_policy.py`
- 按第一节的规则实现:出牌、碰杠胡、定缺、换三张决策
- `rule_policy` 实现 `base_policy` 接口:输入 S2 状态 + legal_mask,输出一个合法动作
- **只在 legal_mask 允许的动作里选**(绝不输出非法动作)
- **验收**:策略对任意合法状态都能输出合法动作;行为符合规则(有缺门必打缺门、能胡则胡等,构造场景测试)

### 任务 3:自对局 `run_selfplay.py`
- 4 个 S3 策略互打,跑 N 局(可配),用 S1 引擎驱动
- 统计:流局率、平均胡牌人数、各番型分布、平均每局得分分布
- **验收**:
  - 能稳定跑完 10000 局不崩、零和守恒仍成立
  - **流局率明显低于之前的贪心策略**(贪心约49%,S3 应更低,目标 <40%,具体看实现)——这是"S3 比贪心会做牌"的证据
  - 各番型正常出现,行为合理

### 任务 4:数据记录 `data_recorder.py`(为 S4 准备)【v4 调整:记协议态,不记张量】
- 自对局时,每个决策点记录三样东西:
  1. **玩家视角的 s2.v4 协议状态(JSON,完美观测)——不是编码后的张量**。原因:S4/S5 必须在"观测退化"后的残缺观测上训练,而退化必须发生在编码之前;存张量就没法事后退化了。退化 + 编码放到 S4 的数据管线里做,同一决策点可退化出多份训练样本,一局数据顶多局用
  2. **玩家选择的动作**(动作索引)+ 该局最终得分(供 S4/S5 用)
  3. **上帝视角标签**(调用 S2 `tile_belief` 的标签生成器):每张未知牌的真实位置(牌墙/哪个对手)、各对手真实听牌状态、各候选弃牌是否真实点炮张。**与输入分开存放,任何情况下不得进入输入特征**——这是 S4 训练 belief 网络的批改答案
- 只记录"对局阶段"的决策(出牌/碰杠胡),开局换牌定缺可单独标记
- **验收**:跑1000局产出数据集,格式正确、可被加载;协议状态通过 s2.v4 schema 校验;标签与上帝视角逐张一致;输入部分全程可见性无泄漏(抽样断言协议里不含任何对手暗牌);演示"同一决策点 → 观测退化 → 编码"的下游流程跑通至少一例

### 任务 5:基准线封装 + 对手池 + 知识注解
- 把 S3 封装成"标准对手",后续 S4/S5 评测时直接用它当对打基准
- **对手池打包(为 S5 前置约束服务)**:把随机策略、贪心策略与 S3 一起封装成统一 `base_policy` 接口的三档标准对手(弱/中/基线)。S5 spec 要求对手池多样化(缓解"自博弈对手 ≠ 真人对手"的分布差),这三档是现成的强度梯度,本阶段顺手交付
- `docs/concepts.md`:注解向听数概念、启发式策略 vs 学习策略的区别、为什么 S3 是基线而非终点、它如何为 S4 提供数据
- **验收**:S3 可作为固定对手被调用;文档清晰

## 四、全局技术约束
- 复用 S1 引擎、S2 状态层(import);S3 不碰引擎规则、不碰状态编码
- 策略是确定性的(同状态同输出)或可控随机(同seed可复现),便于调试与对打
- 向听数计算要正确高效(它会被海量调用,自对局跑很多局)
- 数据记录存 S2 **玩家视角的协议状态**(输入侧绝不能混入上帝视角——S4 学的是"基于可见信息决策";上帝视角只出现在单独存放的标签文件里,供 belief 网络批改用)
- 每模块 pytest 单测;向听数计算是重点测试对象

## 五、执行顺序
任务1(向听数)→ 任务2(规则策略)→ 任务3(自对局,验证流局率下降)→ 任务4(数据记录)→ 任务5(基准封装+注解)

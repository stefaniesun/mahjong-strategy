# S2 编码器 v4 增补规格:候选弃牌特征(把老师的算盘交给学生)

> 完成后需用现有数据重训 S4 并重跑 500 局门禁(见第六节)。

## 一、背景与诊断(为什么改)

**门禁史**(500 局同种子实战,0 号位对 3 个 S3,基线 -0.044±0.39):

| 版本 | 训练配方 | 场均分差 | 胡牌率 | 打牌一致率 |
|---|---|---|---|---|
| v1 | 1000 局 | -2.56 | 26.8% | 62.2% |
| v2 | 5 万局 + encoder.v3(对手弃牌/副露特征) | -2.47 | 33.6% | 74.5% |
| v3 | + DAgger 第 1 轮(2 万局) | **-1.30** | 51.2% | 75.8% |
| v4 | + DAgger 第 2 轮(2 万局) | -1.26 | 48.8% | 76.9% |

关键事实:**DAgger 第 2 轮收益归零**(v3→v4 在置信区间内持平)——"学生走进陌生局面就崩"的分布漂移病已被第 1 轮治愈,剩余 -1.2 分的差距是另一种病。

**病根定位**:S3 老师的出牌决策依据是**每张候选牌打出后的向听数变化 + 有效进张的剩余张数**(`heuristics.py` 的核心逻辑)。而当前编码器(见 `encoding_table()`)只提供原始手牌计数(`own_hand_counts` 27 维),**没有任何候选弃牌的牌理特征**——等于要求一个 256 隐层的小 MLP 从原始计数自行学会向听数的组合计算。它到 76% 一致率即触顶:剩下的错不是"没见过",是"算不动"。

**药方**:把 S3 决策所用的量直接算好、编进输入。模仿从"猜"变成"抄",预期打牌一致率升至 90% 级,门禁过线(-0.5 以内)。

## 二、新增编码区段(追加在现有结构表末尾,offset 806 起)

沿用现有约定:数值归一化;区段带 unknown 标志位与置信度位。所有计算只用**自己的手牌 + 自己的定缺 + 统计层的剩余牌计数**——全部是玩家可见信息,不碰可见性红线。

### 区段 1:`own_hand_shanten`(当前手牌牌理,4 维)

- 当前向听数 / 8;
- 当前有效进张的**剩余张数合计**(对 `useful_tiles(hand, void)` 的每张牌,累加 `remaining_tile_counts` 中的对应值)/ 40,截断到 1.0;
- unknown 标志位 + 置信度位。

### 区段 2:`candidate_discard_features`(候选弃牌特征,27 × 3 + 2 = 83 维)

对 27 种牌各 3 维:

1. **可弃标志**:该牌是否在自己暗手牌中(0/1);不在手中的牌后两维填 0;
2. **打出后向听数** / 8:从手牌移除该牌后重算 `shanten`(计入定缺约束);
3. **打出后有效进张剩余张数** / 40(截断 1.0):移除该牌后的 `useful_tiles` 逐张按 `remaining_tile_counts` 累加——**这正是 S3 的同分决胜指标,必须用同一口径**(剩余张数来自统计层软计数,不是理论 4 张)。

区段末尾:unknown 标志位 + 置信度位。

**共新增 87 维,总维度 806 → 893。**

### 计算与降级规则

- 数据来源:视角玩家自己的 `concealed_hand`(恒为可见)、自己的定缺、`statistics.remaining_tile_counts`;
- `remaining_tile_counts` 为 unknown 时,进张剩余张数退化为按"理论剩余 = 4 − 自己手牌内张数"估算,置信度位相应下调;`concealed_hand` 不可得(异常态)时整段置 unknown、数值全 0;
- **所有对局阶段都计算**(出牌/响应/换三张/定缺阶段一律按当前手牌算,响应阶段的数值对碰杠决策同样有参考价值);
- 复用 `state/hand_analysis.py` 的 `shanten` / `useful_tiles`(两者已有结果缓存,27 次试算的重复子问题会大量命中缓存)。

## 三、必须遵守的约束

1. **只追加,不重排**:0~805 维的现有区段 offset、含义、数值一律不变,用升级前编码器对同一状态的输出做金标准回归;
2. **`ENCODER_VERSION` 升为 `s2.v4.encoder.v4`**;旧 checkpoint(encoder.v3)经 `LearnedPolicy` 加载必须报错拒载(已有校验机制,补专项测试);
3. **性能预算**:单次 `encode_state` 耗时相对 v3 的增幅 ≤ 100%(缓存命中下应远低于此);冒烟时实测报数。若超标,优先检查是否在逐候选重复构造 Hand 对象(应复用计数数组);
4. **可见性红线**:只读自己的手牌;沿用"同可见信息同输出"测试;
5. **确定性**:同协议输入同输出;
6. **不改协议、不改数据、不改 S3/引擎**:纯编码层改动。已有的 5 万局 + 两轮 DAgger 数据(协议级记录)直接可用,**无需重新生成任何数据**。

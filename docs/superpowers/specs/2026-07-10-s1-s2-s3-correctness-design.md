# S1–S3 规则、可见性与策略接口整改设计

## 1. 背景与目标

本次整改修复三条相互依赖的基础链路：

1. S1 对局引擎必须在动作生成和执行层共同保证胡牌优先级、海底禁杠及胡牌上下文正确。
2. S2 玩家视角协议不得泄漏由其他玩家暗手牌推导出的信息，belief 标签必须准确表达未知牌副本位置。
3. S3 策略必须只消费 S2 玩家视角协议与固定动作空间合法掩码，不能直接读取 S1 上帝视角状态。

整改按 S1 → S2 → S3 串行实施。每个阶段采用测试驱动开发，专项测试通过后再运行受影响测试和完整测试套件。

## 2. 范围

### 2.1 S1 规则闭环

- 点炮胡优先于碰、直杠，且不能通过直接调用 `Game.step()` 绕过。
- 所有阶段和响应动作在修改状态前通过统一合法性门禁。
- 牌墙只剩最后一张时，直杠、暗杠和补杠均不可生成、不可执行。
- 显式维护本次摸牌是否为杠后补牌、是否为牌墙最后一张。
- 在真实对局链路接通杠上花、杠上炮、海底捞月和海底炮。

### 2.2 S2 可见性与 belief 标签

- `pending_discard` 与 `pending_rob_kong` 仅暴露公开事件，不输出候选赢家及其相对位置。
- 本玩家是否能胡仅通过共享合法性计算生成的动作及 `legal_mask` 表达。
- tile-location 标签改成每种牌在四个隐藏位置的真实副本计数及归一化分布。
- 监督 mask 由该牌种的未知副本数量决定。
- 对手听牌 oracle 必须传入对应玩家的定缺门。
- 编码器加入各玩家公开定缺信息，使输入具备学习定缺约束的条件。

### 2.3 S3 策略接口迁移

- `Policy.select_action()` 统一接收 `S2ProtocolState` 与固定动作空间 `legal_mask`。
- 内置规则、随机、贪心策略只读取玩家可见事实、统计、belief 和合法掩码。
- 自对局、竞技场和工具仍以 `Game` 为裁判，但不得把完整 `GameState` 交给策略。
- S4 数据记录器和教师策略使用同一玩家视角接口。
- 迁移完成后不保留可继续调用 `GameState` 策略接口的长期双轨设计。

## 3. 不在范围内

- 改变四川麻将番型定义、基础分或封顶配置。
- 新增高级 S3 战术、对手建模或防守策略。
- 为旧 belief checkpoint 提供语义不可靠的兼容训练模式。
- 重新生成大规模正式训练集或重新训练正式模型；代码整改完成后应另行执行。

## 4. S1 设计

### 4.1 统一执行门禁

`Game.legal_actions()` 继续负责生成当前状态下的合法动作。`Game.step()` 在任何状态变更前重新取得动作所属玩家的合法动作，并验证提交动作属于该集合。校验失败时抛出明确规则异常，且状态不得发生变化。

响应队列保持以下优先级：

1. 抢杠胡响应；
2. 点炮胡响应；
3. 碰或直杠响应；
4. 普通出牌、自摸、暗杠或补杠。

当 `pending_winners` 非空时，只有候选胡牌玩家可提交胡或过；其他玩家不能获得或执行碰、杠。执行层门禁是最终保障，不能依赖调用方按顺序调用。

### 4.2 海底禁杠

所有杠动作统一要求牌墙在执行前至少保留两张牌，即 `len(wall) > 1`。该约束同时进入：

- 引擎动作生成；
- S2 共享合法性；
- `Game.step()` 执行校验。

这样既防止正常调用生成非法动作，也防止直接构造动作绕过。

### 4.3 胡牌上下文

引擎状态显式记录当前摸牌上下文，而不是从牌墙当前长度或历史动作临时猜测：

- `after_kong_draw`：当前手牌新增牌是否来自杠后补牌；
- `last_wall_draw`：当前手牌新增牌是否为牌墙最后一张。

杠后补牌时设置 `after_kong_draw=True`；普通摸牌设置为 `False`。摸走牌墙最后一张时设置 `last_wall_draw=True`。玩家弃牌后，自摸上下文结束，但该弃牌保留海底炮及杠上炮所需的公开响应上下文，直至响应窗口结束。

结算分支统一构建 `WinContext`：

- 自摸：`self_draw=True`，并传 `after_kong`、`haidi`；
- 点炮：传弃牌对应的 `after_kong`、`haidi`；
- 抢杠胡：`robbing_kong=True`，不误继承过期摸牌上下文。

上下文只对紧随其后的补牌、自摸或弃牌响应有效，不得泄漏到后续回合。

### 4.4 原子性

`Game.step()` 必须先完成玩家、阶段、响应优先级、动作参数、手牌数量及牌墙约束校验，再修改手牌、牌墙、pending 状态或分数。非法动作失败后至少保证以下字段不变：

- `pending_discard`、`pending_winners`、`pending_rob_kong`；
- 玩家手牌与副露；
- 牌墙；
- 当前玩家与阶段；
- 分数与杠流水。

## 5. S2 设计

### 5.1 玩家视角事件

`from_engine()` 生成的待响应事件只保留公开事实，例如弃牌者、弃牌牌面、补杠者及补杠牌面。删除：

- `winners`；
- `winner_relatives`；
- 其他由非观察者暗手牌计算出的候选响应者集合。

协议安全性不能依赖“当前编码器暂未编码某字段”。任何玩家视角协议字段都必须满足可见性要求。

### 5.2 本玩家合法动作

适配器构造公开协议后，使用 `state.legality.legal_actions(protocol_state)` 计算本观察者合法动作，并通过 `state.action_space` 生成固定长度 `legal_mask`。只有本玩家手牌参与本玩家胡牌合法性判断，不读取其他玩家暗手牌。

训练态保留引擎与共享合法性的交叉验证，要求完美观测下结果一致；玩家协议不携带用于完成交叉验证的上帝视角中间值。

### 5.3 tile-location 标签

对每个牌种建立四个位置：

1. 牌墙；
2. 下家暗手牌；
3. 对家暗手牌；
4. 上家暗手牌。

标签生成器逐个物理副本统计真实位置，输出：

- `counts[27][4]`：各位置未知副本真实计数；
- `distribution[27][4]`：当未知副本数大于零时，按行归一化的真实分布；
- `mask[27]`：该牌种是否存在需要监督的未知副本。

若某牌种未知副本数为零，则其分布填中性值并令 mask 为 0，不计入损失。若计数为负、超过四张或总数不守恒，标签生成立即报错，不静默截断。

belief 网络 tile-location 输出形状仍为 `27 × 4`，但训练目标从单一硬类别改为软目标交叉熵：

\[
L = -\frac{\sum_t m_t \sum_l q_{t,l}\log p_{t,l}}{\max(1, \sum_t m_t)}
\]

其中 \(q\) 为真实副本分布，\(p\) 为模型预测，\(m\) 为牌种监督 mask。

该语义与旧 checkpoint 不兼容，正式模型需要使用新标签重新训练。

### 5.4 听牌标签与定缺

对手听牌 oracle 调用胡牌/听牌分析时必须传入该玩家 `void_suit`。只要玩家仍持有定缺门牌，就不能被标记为合法听牌。

各玩家已公开的定缺门同时进入协议编码，避免标签使用定缺约束而输入缺少对应事实。

### 5.5 旧数据处理

旧 JSON 协议中的候选赢家字段在读取时忽略，不进入协议特征。旧 tile-location 单类别标签不自动转换为副本分布，因为转换会虚构已丢失的副本位置信息。正式训练集应使用修复后的标签生成器重新生成。

## 6. S3 设计

### 6.1 统一策略接口

策略接口语义为：

```text
select_action(protocol_state: S2ProtocolState, legal_mask: Sequence[bool]) -> Action
```

`legal_mask` 长度必须等于固定动作空间长度。策略只能从 mask 允许的候选动作中选择。mask 长度错误或策略返回 mask 外动作均视为策略错误，不自动替换为随机动作。

### 6.2 裁判与策略边界

自对局或竞技场每个决策点执行：

```text
GameState（仅裁判持有）
  → from_engine(player_id)
  → S2ProtocolState
  → legal_actions + legal_mask
  → Policy.select_action(protocol_state, legal_mask)
  → Action
  → Game.step(Action)
```

`GameState` 仅用于推进牌局、结算和生成独立 oracle 标签。策略对象、启发式函数和策略模型入口均不得收到完整引擎状态。

### 6.3 RulePolicy 可用信息

`RulePolicy` 可使用：

- 本玩家可见手牌和副露；
- 各玩家公开弃牌、副露、定缺及胡牌状态；
- S2 手牌分析、候选动作特征、已见牌统计；
- belief 层输出；
- `legal_mask`。

不得使用：

- 对手暗手牌；
- 牌墙顺序或未公开牌；
- 候选赢家 oracle；
- 结算层内部待处理名单。

### 6.4 调用方迁移

同步迁移：

- `policies/base_policy.py`；
- `policies/rule_policy.py` 与其他内置策略；
- `policies/opponent_pool.py`；
- `selfplay/run_selfplay.py`；
- `selfplay/data_recorder.py`；
- `learning/eval/arena.py`；
- 使用策略的工具脚本和测试。

若迁移期间需要适配器，适配器只能位于裁判调用边界，负责从引擎生成 S2 协议；不能把 `GameState` 重新暴露给策略。全部调用方完成后删除旧接口。

## 7. 测试设计

### 7.1 S1 专项测试

1. 有 `pending_winners` 时，其他玩家不获得碰或杠。
2. 直接提交碰或杠被 `step()` 拒绝，pending 状态、手牌、牌墙和分数不变。
3. 牌墙剩一张时，直杠、暗杠、补杠都不出现在合法动作中。
4. 直接提交上述三类杠均被拒绝且状态不变。
5. 七对杠上花按附加 1 番结算；目标示例为胡家 `+27`，其余三家各 `-9`。
6. 牌墙最后一张自摸触发海底捞月。
7. 摸走最后一张后弃牌，其他玩家点炮触发海底炮。
8. 普通后续回合不继承杠后或海底上下文。
9. 抢杠胡不错误继承杠上花或海底上下文。

### 7.2 S2 专项测试

1. 四个观察者协议均不包含候选赢家及相对位置。
2. 两个公开信息相同、其他玩家暗手牌不同的状态，协议输入与编码相同。
3. 本玩家可胡时，合法动作仍正确包含胡；其他玩家是否可胡不影响本玩家协议。
4. 同一种未知牌同时存在于牌墙和对手手中时，所有副本计数均保留。
5. `distribution` 行归一，`mask` 与未知副本数一致。
6. 无未知副本的牌种不参与 tile-location 损失。
7. 持有定缺门牌的玩家不会被标为听牌。
8. 编码器能区分各家的公开定缺门。
9. 标签守恒异常会明确失败。

### 7.3 S3/S4 专项测试

1. 所有内置策略仅通过新接口调用，并始终返回 mask 内动作。
2. `legal_mask` 长度错误时策略拒绝执行。
3. 公开信息相同但隐藏牌不同的两个状态，确定性策略得到相同输入并作出相同决策。
4. 自对局、arena 和回放导出冒烟通过。
5. 数据记录中的输入协议无隐藏信息，oracle 只存在于独立标签字段。
6. belief 数据集可构建新分布标签，单批训练损失有限且可反向传播。
7. policy 数据集和单批训练保持可用。

## 8. 实施和提交顺序

### 阶段一：S1

先增加失败测试，再实现统一门禁、海底禁杠和胡牌上下文。运行 S1 专项测试、S1 全部测试及完整套件。验证通过后独立提交并推送。

### 阶段二：S2

先增加泄漏、分布标签和定缺失败测试，再修改协议、标签、损失、mask 和编码。运行 S2/S4 belief 专项测试及完整套件。验证通过后独立提交并推送。

### 阶段三：S3

先把策略接口测试改为 S2 协议与 mask，再迁移策略和所有调用方。运行 S3、自对局、arena、S4 policy 专项测试及完整套件。验证通过后独立提交并推送。

## 9. 验收标准

1. 胡牌候选存在时，碰或杠既不能生成也不能执行，非法调用不改变状态。
2. 牌墙剩一张时，任何杠均不可生成、不可执行。
3. 七对杠上花按每家支付 9、胡家获得 27 结算。
4. 海底捞月和海底炮均能从真实对局链路触发。
5. 任意玩家视角协议不包含其他玩家是否能胡的信息。
6. 同种未知牌分布于多个位置时，标签保留所有副本并正确归一化。
7. 听牌 oracle 遵守玩家定缺门，编码包含公开定缺信息。
8. 内置策略、自对局和 S4 教师策略只消费 S2 玩家视角状态与 `legal_mask`。
9. S1–S4 相关测试及完整测试套件全部通过。
10. 三个阶段分别形成可审查提交并推送当前分支；若工作区未配置 Git 仓库或远端，则明确报告该环境阻塞，不伪造提交结果。

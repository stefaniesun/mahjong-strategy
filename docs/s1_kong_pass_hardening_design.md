# S1 过庄与杠后转水硬化设计

## 目标

在现有完整对局闭环基础上，补齐第一批会直接影响合法动作与奖励分数的复杂规则：过庄锁定、杠后点炮的杠上炮番、以及有钱杠的杠转水结算。范围暂不包含完整碰/杠动作链与流局退杠钱，避免一次改动过大。

## 范围

### 本次实现

1. 过庄锁定
   - 玩家面对可点炮胡机会选择 `PASS` 后，进入过庄锁定。
   - 锁定持续到该玩家下次摸牌时解除。
   - 锁定期间普通点炮胡不进入合法动作。
   - 若当前可胡番数大于当初放弃番数，则允许胡并视为更大番解锁。
   - 自摸不受过庄锁定影响。

2. 杠后点炮上下文
   - `GameState` 记录最近一次“杠后紧接弃牌”的上下文。
   - 任意杠后紧接弃牌点炮，都给胡牌者 `after_kong=True`，用于杠上炮 +1 番。
   - 只有有钱杠（直明杠、暗杠）参与转水；补杠不转水。
   - 杠后弃牌未点炮或响应窗口结束后，转水窗口关闭。

3. 杠转水结算
   - 通过 `GangRecord` 记录有钱杠的付款人、金额、总额与转水状态。
   - 点炮胡时，若存在可转水杠，则追加“杠者 → 胡牌者”的转水流水。
   - 一炮多响时，每个胡牌者各获得一份完整转水金额。
   - 每一步保持分数零和。

### 不在本次实现

- 完整碰/直明杠/暗杠/补杠动作链。
- 抢杠胡。
- 连续杠完整交互。
- 荒庄查叫与未听牌退杠钱。

这些留到后续批次，避免把动作协议、结算与荒庄审计一次混改。

## 代码设计

### `engine/gang.py`

扩展 `GangRecord`：

- `gang_id`
- `gang_player`
- `gang_type`
- `payments`
- `transferred_to`
- `transfer_count`
- `refunded`

保留 `total_amount` 计算属性。

### `engine/state.py`

新增：

- `gang_records`
- `next_gang_id`
- `last_transferable_gang_id`
- `after_kong_discard_player`

这些字段只描述结算上下文，不负责执行动作。

### `engine/settlement.py`

新增两个小函数：

- `record_kong_payment(...)`：执行杠即时结算并返回 `GangRecord`。
- `apply_kong_transfer(...)`：执行杠转水，追加转水标记并保持零和。

保留 `apply_kong_payment(...)` 兼容既有测试。

### `engine/game.py`

调整点炮响应：

- 检测点炮可胡时应用过庄过滤。
- 玩家对可胡牌 `PASS` 时记录 `passed_hu_lock` 与 `passed_fan`。
- 玩家摸牌时解除自己的过庄锁定。
- 点炮结算时根据 `last_transferable_gang_id` 设置 `after_kong=True` 并执行转水。
- 弃牌响应完全结束后关闭杠后转水窗口。

## 测试设计

新增或扩展测试覆盖：

1. 放弃点炮后，低番/同番点炮胡被锁。
2. 放弃点炮后，下次摸牌解除锁定。
3. 锁定期间更大番点炮胡可进入合法动作。
4. 杠后点炮会给胡牌者杠上炮 +1 番。
5. 有钱杠后点炮会把完整杠钱转给胡牌者。
6. 补杠后点炮只计杠上炮番，不发生转水。
7. 一炮多响时，每个胡牌者各获得一份完整转水。
8. 所有专项结算后保持零和。

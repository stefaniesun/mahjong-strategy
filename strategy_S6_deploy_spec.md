# S6 需求规格:验证与部署对接(从"模拟器里的强者"到"真实牌桌上的助手")

> 策略 AI 项目第六阶段,也是最后一个阶段。S1~S5 已交付:规则引擎、状态层(s2.v4)、规则基线、belief 网络(读牌)、策略网络(决策),且已通过双赛道(完美/残缺观测)验收。
> 本阶段**不训练任何新模型**。核心任务:把两个模型组装成真实可用的辅助系统,接入视觉线,用三级验证(回放 → 影子 → 实测)回答最终问题——**"它在真实牌桌上到底行不行"**,并建立发现问题后回流训练的迭代闭环。
> 交给 AI 编码助手执行。Python 3.10+,依赖少,有单测。

---

## 设计定位(给执行者的关键认知)

**S6 是 sim2real 的最终考场。** 前面所有阶段的核心假设是:"在退化观测上训练的模型,能在真实残缺输入下正常工作"。S4/S5 用模拟的退化验证过这个假设,但模拟的噪声画像终究是我们**猜**的真实分布。S6 用真实数据校对这个猜测,猜错的部分回流修正——这不是一次性验收,是一个循环。

**部署红线(继承 S2 v4,必须死守)**:
- 部署运行时**不得 import S1 引擎的对局推进逻辑**。真实牌局由现实世界驱动,运行时只依赖:S2 协议/编码/`tile_counting`/`legality.py`(共享纯函数)+ 两个模型文件;
- 可见性红线照旧:运行时只处理用户合法可见的信息(这在部署时是物理成立的——摄像头本来就拍不到对手手牌,但代码层面依然要保证不存在任何依赖隐藏信息的路径);
- **建议必须诚实**:概率来自校准过的 belief 网络,置信度低就明说低,前提假设(如"假设你未过庄")必须随建议一起呈现。辅助工具的生命线是用户信任,一次胡说八道的代价大于十次保守。

**三级验证的逻辑**(成本递增、真实度递增,逐级放行):
1. **回放验证**:拿真实牌局录像离线跑,零风险,先过这关;
2. **影子测试**:真人正常打牌,系统旁观并默默给建议,记录"建议 vs 真人选择 vs 实际结果",不干预牌局;
3. **实测**:真人参考建议打牌,统计真实战绩。

## 一、系统形态

```
摄像头/截图 → 视觉识别模型(视觉线交付,不在本 spec 范围)
                  ↓ VisionEvent 事件流(S2 任务 8 定稿的 schema:事件+置信度+备选)
          ┌─────────────────────────────────────┐
          │  S6 辅助运行时(本阶段交付)            │
          │  vision_bridge:事件流 → s2.v4 协议    │
          │   (含 108 张守恒调和、矛盾检测)        │
          │  tile_counting:数牌(写死的,满分)      │
          │  belief 网络:读牌(概率)               │
          │  legality:哪些动作合法(共享纯函数)     │
          │  策略网络:决策                        │
          │  suggestion:组装建议                  │
          └─────────────────────────────────────┘
                  ↓
   给用户的建议:推荐动作 + 备选 + 理由 + 概率 + 前提假设 + 置信度
```

### 建议的呈现格式(产品核心,定死结构)

每次轮到用户决策,输出一个结构化建议对象:

- **推荐动作** + 1~2 个备选动作(按策略网络概率排序);
- **危险提示**:候选弃牌的放炮概率(belief 输出,校准过的真概率),超过阈值的牌明确标红;
- **读牌摘要**:关键未知牌的位置估计(如"最后一张四万约 70% 在牌墙")、各对手听牌概率;
- **前提假设**:凡建议依赖了 unknown 字段的宽松假定(`conditionally_legal` 等),逐条列出(如"假设你本轮未放弃过胡牌");
- **整体置信度**:观测完整度摘要(中途接入、近期识别矛盾次数等折算),低于阈值时建议降级为"信息不足,仅供参考"并说明缺什么。

## 三、任务清单

### 任务 1:推理运行时 `model_bundle.py` + `assistant.py`

**做什么**:两个模型 + 纯规则模块组装成独立运行时。

- `model_bundle`:打包 belief/策略 checkpoint + 编码结构表版本 + 退化画像版本 + 规则配置;加载时自检指纹匹配(防止模型和编码器版本错位——这是部署最阴险的静默错误);
- `assistant`:消费协议状态,增量更新(每来一个事件只做增量计算,不整局重算);输出建议对象;
- **部署红线自检**:静态检查部署代码路径的 import 白名单(S2 协议/编码/计数/legality + torch),出现引擎对局逻辑即 CI 失败;
- **验收**:同事件流同建议(确定性);**延迟预算达标并报数**——单决策点端到端(事件已到达 → 建议产出)≤ 200ms(其中模型推理应在个位数 ms,大头留给状态更新),普通笔记本 CPU 实测;import 白名单测试通过。

### 任务 2:视觉桥接 `vision_bridge.py` + 会话管理 `session.py`

**做什么**:把真实视觉输出接进 S2 任务 8 定稿的事件 schema,并管理一局的生命周期。

- `vision_bridge`:视觉模型原始输出 → `VisionEvent`(置信度、备选候选照搬);快照事件支持(中途接入时对牌池/副露一次性识别);守恒调和与矛盾检测走 S2 `from_vision` 通路;
- `session`:开局检测、**中途接入**(从任意时刻的快照冷启动,`observation_start` 正确设置)、识别矛盾达到阈值时的恢复策略(提示用户人工校正指定牌 → 修正后继续)、一局结束与换局重置;
- **人工校正通道**:用户可随时手动修正某个识别结果(改牌面/补漏),修正以 observed(置信度 1.0)身份覆盖原 estimated 观测并触发重调和——这是视觉不完美时的兜底,必须有;
- **验收**:用构造事件流 + 至少一段真实视觉输出联调:开局/中途接入/含错误识别/含人工校正四种情形全部走通;矛盾检测触发后恢复流程正确;换局后无状态残留。

### 任务 3:回放验证 `replay_eval.py`(三级验证之一,先过)

**做什么**:拿真实牌局录像(经视觉线转成事件流)离线回放,系统在每个决策点给建议,离线评审。

- 输入:若干局真实对局的完整事件流(我负责提供录像,视觉线负责转换);
- 输出评审材料:每个决策点的"牌面局势 + 系统建议(含理由/概率)+ 真人实际选择",按局组织,供人工逐点评审;
- **真实数据校准核对(关键)**:汇总所有"系统说某张放炮概率为 p"的样本,对照真实结果统计实际放炮频率,画可靠性对照——**模拟器里校准好的概率,在真实分布上还准不准,这是 sim2real 假设最直接的检验**;
- **验收**:≥20 局真实牌局回放全程无崩、无非法建议;校准对照报告产出;评审材料我(人类)看得懂、能逐点打分。

### 任务 4:影子测试 `shadow_mode.py`(三级验证之二)

**做什么**:真实牌局进行中,系统实时旁观并生成建议,**只记录不展示**(或仅测试者可见),不影响打牌。

- 记录每个决策点:建议、真人选择、是否一致、该局最终结果;实时性能数据(延迟分布、矛盾次数、置信度分布);
- 统计:建议-真人一致率(分阶段:开局/中盘/尾盘)、不一致点的复盘清单(可能是系统错,也可能是真人错——逐条人工判);中途接入场景专项(故意从中盘开机);
- **验收**:≥30 局影子运行全程稳定(零崩溃、延迟达标率 >99%);产出一致率报告与不一致复盘清单;中途接入专项 ≥5 局行为正常。

### 任务 5:实测 + 报告 `field_report.py`(三级验证之三)

**做什么**:真人参考建议打牌,统计真实战绩;所有真实数据汇总成 S6 验证报告。

- 实测协议:固定局数(建议 ≥50 局起步)、记录每局得分、关键决策点采纳/不采纳标记;
- 对照基准:同一测试者不用辅助时的历史战绩(或影子阶段的战绩)作参照——样本量有限,结论谨慎表述(给区间不给断言);
- **失败案例归档**:建议明显不合理的决策点逐个归档(牌面、系统输入、各模块输出),按根因分类:视觉识别错 / 协议调和错 / belief 读错 / 策略选错 / 呈现误导,供任务 6 回流;
- **验收**:实测报告产出(战绩统计 + 采纳率 + 失败案例分类账);每个失败案例能定位到具体模块(这是分层架构的价值兑现点)。

### 任务 6:sim2real 迭代闭环 `noise_profiler.py`(让系统能持续变好)

**做什么**:把真实世界暴露的问题,变成回流训练的具体动作。这是 S6 的长期价值所在。

- `noise_profiler`:从影子/实测数据统计**真实的视觉误差画像**——混淆矩阵(哪张牌实际被认成哪张、频率)、漏检率、矛盾模式——输出为 S2 观测退化模块的配置格式;
- **回流规则(写成明确的触发条件)**:
  1. 真实误差画像与训练用画像差异超阈值 → 更新 S2 退化配置 → 重跑 S4(belief 重训 + 重验收)→ 视幅度决定是否重跑 S5;
  2. 真实校准核对(任务 3)显著劣化 → 优先排查视觉画像差异,其次考虑 belief 重训;
  3. 失败案例集中在"策略选错" → 归入 S5 的下一轮训练素材与评估局面;
  4. 集中在"视觉识别错" → 提给视觉线,不动策略侧;
- 版本管理:每次回流产生新 model_bundle,新旧版本在回放集 + 影子模式上 A/B 对比后才准上岗;
- **验收**:noise_profiler 从真实数据产出合法的退化配置;回流触发条件文档化;演示一次完整小循环(真实画像 → 更新配置 → 重训 belief → 新 bundle 回放对比)。

### 任务 7:知识注解 `docs/concepts.md`

- sim2real gap 的最终形态:为什么模拟验收过还要三级真实验证;为什么校准要在真实数据上重新核对;
- 三级验证的成本/风险/信息量权衡;失败案例按模块归因为什么可行(分层架构的回报);
- 迭代闭环:辅助系统不是"训完就完",是"部署 → 测量 → 回流"的循环;
- **验收**:读完能说清"上线不是终点"以及每类真实世界问题该回流到哪一层。

## 四、全局技术约束

- Python 3.10+;运行时依赖最小化(numpy + torch + S2 的纯规则模块);**import 白名单 CI 强制**(部署路径无引擎对局逻辑);
- 建议诚实红线:概率必须来自校准链路;置信度不足必须明示;前提假设必须随建议呈现;
- 延迟预算:决策点端到端 ≤200ms(CPU);全程增量计算,禁止整局重算;
- 所有真实数据(回放/影子/实测)本地留存、格式与 S2 协议兼容,可直接进回流管线;
- 确定性:同事件流同建议;model_bundle 指纹自检;
- 每模块 pytest 单测;守恒调和、人工校正覆盖、中途接入冷启动、import 白名单是重点测试对象。

## 五、执行顺序

任务 1(运行时)→ 任务 2(视觉桥接+会话)→ 任务 3(回放验证,过关才继续)→ 任务 4(影子测试)→ 任务 5(实测+报告)→ 任务 6(迭代闭环)→ 任务 7(注解)

(任务 6 的 noise_profiler 可在任务 4 有数据后即开始;任务 3/4/5 之间是放行关系,前一级发现严重问题先回流修复再进下一级。)

## 六、我(人类)负责

- 提供真实牌局录像(回放验证用)与安排影子/实测的牌局场次;

## 七、本阶段成功标志

**运行时延迟达标、零非法建议、版本指纹自检有效;回放验证通过且真实数据校准核对合格;影子测试 ≥30 局稳定、不一致复盘完成;实测 ≥50 局产出战绩报告与按模块归因的失败案例账;noise_profiler 能从真实数据产出退化配置并演示一次完整回流小循环。**

至此整条线闭环:S1 引擎(练习场+裁判+标签工厂)→ S2 状态层(残缺信息的语言)→ S3 规则基线(陪练+数据工厂)→ S4 监督学习(读牌+冷启动)→ S5 强化学习(真正的牌力)→ S6 部署验证(真实世界的考场与回流泵)。往后的常态是 S6 的循环:**部署 → 测量 → 回流 → 更强**。

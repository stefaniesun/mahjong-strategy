# S5 本地训练控制台规格:双机长训 + 手动启停界面

> 交给 AI 编码助手执行的独立文档。目标:用现有两台机器搭建 S5 长期训练的运行体系,
> 提供一个**浏览器访问的控制台界面**——手动控制训练开始/暂停,展示训练状态与评估曲线。
> 前置:strategy_S5_prep_spec.md 已完成(RL 基建适配 893 维、v5 三件套冒烟通过、断点续训已验证、rollout 吞吐已实测在案)。

---

## 一、硬件与角色(已定,不要更改)

| | 机器 A(主力 PC,Windows) | 机器 B(Mac mini 2012,Ubuntu) |
|---|---|---|
| 配置 | 至强 E3-1231v3,4核8线程,16GB | i5 双核4线程,16GB |
| 开机习惯 | **仅空调时段**(晚间),受控启停 | **24 小时常驻** |
| S5 角色 | **训练机**:rollout + PPO 学习器 | **控制台宿主 + 评估机**:网页界面、checkpoint 收集、竞技场评测 |

**为什么控制台放 B**:B 常年在线,界面随时打得开;A 没开机时也能看历史曲线和评估结果。A 开机后其代理自动上线,界面上的"开始训练"按钮即刻可用。

## 二、架构总览

```
 你的浏览器(电脑/手机,局域网)
        │  http://<机器B>:8500
        ▼
 ┌─ 机器 B(24h)────────────────────────────┐
 │ console 服务(FastAPI + 静态单页)          │
 │  ├─ 状态聚合:A 代理在线则代理转发           │
 │  ├─ 评估队列:发现新 checkpoint → 竞技场评测 │
 │  ├─ 数据存储:metrics/evals 均为 JSONL 文件  │
 │  └─ checkpoint 仓库(从 A 拉取归档)          │
 └───────────────┬──────────────────────────┘
                 │ 局域网 HTTP(固定IP+共享token)
 ┌─ 机器 A(空调时段)─────────────────────────┐
 │ agent 服务(FastAPI)                        │
 │  ├─ /start /pause /status                   │
 │  ├─ 训练进程管理(spawn train_rl / 优雅停)   │
 │  ├─ 心跳与指标上报(tail 训练进程的指标文件)  │
 │  └─ 安全保护:CPU温度>85°C 自动暂停并告警    │
 │       └── rl/train_rl.py(真正的训练进程)    │
 └─────────────────────────────────────────┘
```

技术约束:全 Python(FastAPI + uvicorn),前端为**单个静态 HTML+JS 页面**(图表用本地 vendored 的 Chart.js 或手写 SVG,禁止任何 CDN 依赖——离线可用);存储一律 JSONL 文件,**不引入数据库、不引入 docker、不引入前端构建工具链**。局域网使用,安全性 = 固定 token(配置文件)+ 只绑局域网,不做用户系统。

## 三、核心语义(必须精确实现)

### 训练控制状态机(机器 A)

```
IDLE ──start──▶ RUNNING ──pause──▶ PAUSING ──(存档完成)──▶ PAUSED ──start──▶ RUNNING
                RUNNING ──(温度>85°C / 磁盘<10G)──▶ PAUSING(自动,界面标注原因)
                任意状态 ──(训练进程异常退出)──▶ ERROR(界面红色,显示日志尾部)
```

- **start**:从最新 checkpoint 续训(`--resume`,含优化器/对手池/课程状态——S5-prep 已验证的机制);无 checkpoint 则从 v5 三件套冷启动;
- **pause 必须是优雅的**:向训练进程发停止信号(建议:写 stop 标志文件,训练循环每个 update 周期检查)→ 训练进程完成当前更新、完整存档、自行退出 → 状态转 PAUSED。**禁止 kill -9 式暂停**;
- 断点续训正确性沿用 S5-prep 的验收标准:暂停→恢复后训练曲线必须衔接;
- A 关机重启后 agent 随系统自启(注册为计划任务/启动项),状态恢复为 PAUSED/IDLE,等待手动 start——**不自动开始训练**(用户手动控制是本设计的核心诉求)。

### 指标流水(训练进程 → 界面)

- 训练进程每个 update 周期追加一行 JSONL(`metrics.jsonl`):时间戳、总局数、吞吐(局/分)、PPO loss、熵、KL、价值损失、当前课程阶段、对手池规模、学习者当前退化画像比例;
- agent-A 提供 `/status`(当前状态机状态 + 最近 N 行指标 + CPU 温度/内存/磁盘);
- B 的 console 定期(如 30s)轮询 A;A 离线时界面显示"训练机离线",历史数据照常可看。

### 评估流水(机器 B,独立自转)

- 训练进程每存一个 checkpoint,agent-A 在 `/checkpoints` 列出;B 每 10 分钟轮询,发现新档即拉取归档;
- B 的评估队列对新 checkpoint 依次跑:**快评**(100 局 vs 3×S3,~20 分钟,B 机速度)出趋势点;每天最新档加跑**正评**(500 局同种子门禁,seed=90000,与 S4 门禁同口径)出正式点;
- 结果追加 `evals.jsonl`:checkpoint 名、局数、场均分差、CI95、胡牌率、对 S3 胜率;
- B 慢没关系(评估赶不上训练就跳过中间档,**始终优先评最新档**)。

### 界面(单页,分四块)

1. **控制区**:大按钮 开始/暂停 + 当前状态灯(含自动暂停原因);训练机健康(温度/内存/磁盘);
2. **训练曲线**:总局数、吞吐、loss/熵/KL(最近 24h / 全程 两档);
3. **强度曲线(最重要的图)**:评估分差 vs S3 随训练局数的变化,标注 -0.044 基线横线与 0 线——这是"S5 有没有在变强"的直观答案;
4. **日志区**:最近事件(启停记录、自动暂停、新 checkpoint、评估完成)+ ERROR 时的训练日志尾部。

## 四、任务清单

### 任务 1:训练进程的控制接口改造(`rl/train_rl.py` 最小改动)

- 支持 stop 标志文件优雅退出(完成当前 update → 完整存档 → 退出码 0);
- 每 update 周期追加 metrics.jsonl(字段见上);
- 温度传感由 agent 负责,训练进程不管;
- **验收**:模拟短训中途置 stop 标志,进程优雅退出且存档完整;续训后曲线衔接(复用 S5-prep 的衔接断言);metrics.jsonl 字段齐全。

### 任务 2:机器 A 代理(`console/agent_a.py`)

- FastAPI:`/start` `/pause` `/status` `/checkpoints` `/checkpoints/{name}`(文件下载)`/log_tail`,全部要求 token;
- 进程管理:spawn/监视训练进程,实现第三节状态机;异常退出转 ERROR 并保留日志;
- 温度守护(LibreHardwareMonitor 或 WMI 读数,>85°C 触发优雅暂停)与磁盘守护(<10G 暂停);
- Windows 开机自启(计划任务),但**不自动 start 训练**;
- **验收**:状态机全路径测试(含自动暂停触发的模拟);断电重启后 agent 自愈为待命态。

### 任务 3:机器 B 控制台(`console/server_b.py` + `console/static/index.html`)

- FastAPI 聚合服务 + systemd 常驻;
- checkpoint 轮询拉取与归档(保留最近 N 个 + 每日里程碑,磁盘上限保护);
- 评估队列(调用现有 `learning/eval/arena.py`,快评/正评策略见第三节;B 机 torch CPU 环境沿用已配好的);
- 单页前端(vendored 图表库或手写 SVG,零外部依赖);
- **验收**:A 在线/离线两种状态下界面均正确;伪造几个 checkpoint 验证评估队列勤最新、跳中间;evals.jsonl 与界面曲线一致;手机浏览器可用(响应式基本可读即可)。

### 任务 4:联调与演练

1. **全链路演练**:界面 start → A 真跑一段短训 → 界面出现指标曲线 → B 拉到 checkpoint 并出快评点 → 界面 pause → A 优雅存档 → 再 start 续训曲线衔接;
2. **异常演练**:训练进程被手动杀死 → 界面 ERROR + 日志;A 直接关机 → B 界面显示离线且历史可看;温度守护模拟触发;
3. **一份 `console/README.md`**:两台机器各自的部署步骤(A:计划任务;B:systemd)、token 配置、端口、故障排查三板斧;
- **验收**:演练清单逐项通过;README 按步骤在两台机器上从零部署成功。

## 五、全局约束

- **训练正确性红线**:控制台是"外挂",不得改变训练本身的任何数值行为——唯一允许的训练侧改动是任务 1 的停止标志与指标输出;暂停/恢复的曲线衔接必须有断言;
- 评估口径红线:B 的正评必须与 S4 门禁完全同口径(同 seed、同对手、同局数),历史数据才可比;
- 全部代码入 git(console/ 目录),部署配置(IP、token、端口)走配置文件且列入 .gitignore,提交模板文件;
- 依赖最小:FastAPI + uvicorn + 现有栈,前端零构建零 CDN;
- **不做**(防发散):不做用户系统/HTTPS/公网访问;不做跨机分布式训练(B 不参与 rollout);不做移动 App;不做训练超参的界面调参(命令行/配置文件改,界面只读展示);不引入数据库与消息队列。

## 六、完成标志

**在手机/电脑浏览器打开 B 的页面:能一键启停 A 上的 S5 训练(优雅暂停、续训曲线衔接);能看到吞吐/loss/熵/KL 实时曲线与"分差 vs S3"强度曲线(B 自动评估产出);A 离线时历史照常可看;温度/磁盘守护有效;两机部署文档可复现。** 此后 S5 长训的日常就是:晚上开 A → 界面点开始 → 睡前点暂停(或让守护自动管),白天随时掏出手机看强度曲线爬坡。

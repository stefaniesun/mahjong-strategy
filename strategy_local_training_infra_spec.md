# 本地训练基建规格:双机低成本训练方案(支撑 S4/S5)

> 横切基建文档,不是新阶段。它回答一个问题:**如何用现有的两台旧机器 + 几百元硬件投入,在本地完成 S4 监督训练与 S5 强化学习,不依赖云。**
> 交给 AI 编码助手执行。执行前先通读 strategy_S4_supervised_spec.md 与 strategy_S5_rl_spec.md,本文档是它们的算力落地方案。
> 云方案(约 ¥18/天租 32 核)保留为后备,本文档末尾给出"何时放弃本地转云"的判据。

---

## 一、硬件清单与角色分工(已定,不要更改)

### 机器 A:主力 PC(Windows)

| 项 | 规格 |
|---|---|
| CPU | Intel Xeon E3-1231 v3,4 核 8 线程,3.4GHz(2014 年,LGA1150 平台,**CPU 无升级空间**) |
| 内存 | 8GB DDR3(**计划升级到 32GB**,见任务 0) |
| 显卡 | GTX 1050 Ti 4GB(Pascal,新版 PyTorch 支持收窄,必要时装 2.1~2.4 + cu118,或纯 CPU 训练) |
| 硬盘 | 500GB SSD,仅剩约 100GB 空闲(需清理/扩容) |
| 环境约束 | **夏季室温可达 40°C,只允许在开空调时段满载**(约每晚 6~8 小时) |

**角色**:开发机 + S4/S5 训练主机(仅空调时段)+ 冒烟测试场。

### 机器 B:Mac mini(Late 2012,已装 Ubuntu)

| 项 | 规格 |
|---|---|
| CPU | Intel Core i5 2.5GHz,2 核 4 线程(笔记本级,单核约为机器 A 的 6~7 成) |
| 内存 | 16GB DDR3 |
| 功耗 | 35W 级,**不怕高温,可 24 小时运行** |

**角色**:全天候数据工厂(离线自对局产数据)+ S5 阶段专职评估机(跑竞技场)。总吞吐约为机器 A 的 25~30%,但可全天出勤,实际日产能贡献接近机器 A 的一倍夜班。

**明确不做**:S5 在线采样的跨机分布式(actor-learner 权重同步)。理由:为约 30% 吞吐编写和调试跨平台分布式代码不划算。两台机器只做"零通信并行"——按种子区间分工、产物文件合并。

## 二、实测性能基线(2026-07,机器 A 实测,后续优化以此为对照)

| 指标 | 实测值 |
|---|---|
| RulePolicy 4 人自对局 | **596 ms/局**(纯 Python),平均 138 步/局 |
| 随机动作驱动 + 学习者座位协议转换+先验belief+编码 | **298 ms/局**,学习者每局约 60 个决策点 |
| 全测试套件 | 149 个 pytest 全过,约 45s |
| 零和守恒 | RulePolicy 2000 局 + 随机 520 局全过,legality 与引擎 6.4 万决策点零不一致 |

由此估算(优化前):S5 采样约 0.7~1 秒/局/核 → 机器 A 8 线程约 3 万局/晚(8h),500 万局需约 20 个夜晚——**太慢,必须先做任务 1 的引擎提速**。

## 三、任务清单

### 任务 0:硬件与环境准备(人类为主,AI 辅助核对)

1. **内存升级(必做,¥150~250)**:机器 A 加装二手 DDR3 1600 台式机内存至 32GB(先开机箱确认插槽数与现有条数)。8GB 下多进程采样会 OOM,这是唯一值得花钱的硬件;
2. **散热维护(必做,¥20)**:机器 A 清灰 + CPU 换硅脂。训练是连续满载,十年机龄散热器不维护会降频/死机;
3. **磁盘(视情况)**:清出 ≥50GB;不够则加 1TB SATA SSD(约 ¥350)专放训练数据与 checkpoint;
4. **机器 B 环境**:Ubuntu 上装 Python 3.10+,同步项目代码,跑 `pytest -q` 必须 149 个全过;
5. **跨平台对拍(必做)**:两台机器用相同种子各跑 10 局 `run_selfplay_game`,最终得分必须逐分一致(引擎是确定性的,不一致即环境问题,先修再继续)。

**验收**:机器 A 内存 ≥32GB 且 8 进程并行采样不 OOM;机器 B 测试全过 + 对拍逐分一致。

### 任务 1:引擎热点 Cython 提速(本方案成败的关键)

**做什么**:把引擎热点函数编译为 C 扩展,目标整体提速 ≥10 倍(596 ms/局 → ≤60 ms/局)。

- 先 profile(cProfile/py-spy)定位热点,预期集中在:`win_check`(胡牌判定)、`fan_calc`(算番)、`actions`/`legal_actions`(合法动作生成)、`hand` 计数操作,以及 S2 的 `encoder`/`hand_analysis`(向听数);
- 优先级按 profile 实际占比排,逐个 Cython 化(加类型声明、去对象开销;手牌 27 维计数数组可用 C 数组/内存视图);
- **行为不变是红线**:每改一个函数,全套 149 测试必须过;另做"新旧实现对拍"——同种子 500 局,逐局逐分一致后才允许替换;
- 保留纯 Python 回退路径(环境变量开关),Cython 编译失败时仍可运行(机器 B 的 Ubuntu 也要能编译,setup 脚本跨平台);
- 不追求一步到位:先拿下 profile 前三名,实测提速,够用即停。

**验收(必须给数字)**:RulePolicy 自对局 ≤60 ms/局(≥10 倍);149 测试全过;新旧对拍 500 局逐分一致;两台机器都能编译运行。达标后重算吞吐账并更新本文档第二节基线表。

### 任务 2:双机数据工厂(S3/S4 数据生成的并行化)

**做什么**:把数据生成按种子区间切给两台机器并行,产物可靠合并。

- 封装数据生成入口:`generate_data --seed-start N --seed-end M --out data/part_X.jsonl.gz`,**输出一律 gzip 压缩**(协议 JSON 不压缩会到几十 GB,机器 A 磁盘扛不住);
- 种子分配按能力比例(约 A:B = 3:1),两台各自独立跑,零通信;
- 合并工具:校验各分片种子区间无重叠无遗漏、schema 抽样校验、合并为最终数据集并生成清单(局数、决策点数、文件指纹);
- 断点友好:按批落盘(如每 500 局一个分片),中断后从下一批继续,不重跑已完成分片;
- 机器 B 常驻运行(nohup/systemd),机器 A 仅空调时段跑。

**验收**:两机并行产出 5 万局数据;合并清单种子连续无重复;抽样通过 s2.v4 schema 校验与可见性检查;压缩后总体积在磁盘预算内。

### 任务 3:训练的暂停/恢复纪律(适配"空调时段"作息)

**做什么**:让 S4/S5 训练完全适配"每晚开工、白天停机"的使用方式。

- checkpoint 间隔 ≤30 分钟(S5 的 checkpoint 按 spec 含优化器/对手池/课程状态);意外断电最多损失半小时;
- 优雅停止:Ctrl+C 触发"先存档再退出";`--resume` 从最新存档继续;
- **时间窗调度**:训练脚本支持 `--schedule 20:00-07:00`,窗口外自动存档休眠、窗口内自动恢复(机器 A 用);机器 B 不设窗口;
- **恢复正确性验收**:同一训练任务"连续跑 2 小时"与"跑 1 小时→停→恢复→再跑 1 小时",损失曲线与评估指标必须衔接一致(允许浮点级差异);这本就是 S5 spec 的要求,在本地方案里提级为必测;
- 温度保险:训练循环内周期读取 CPU 温度(Windows 用 LibreHardwareMonitor 接口或 WMI),超阈值(如 85°C)自动存档暂停并告警——防止空调故障时硬件受损。

**验收**:中断/恢复对拍通过;时间窗调度按配置生效;高温自动暂停可被模拟触发。

### 任务 4:双机分工跑 S4/S5(执行编排)

**S4(预计:数据一晚 + 训练一晚)**:
- 数据:任务 2 双机并行产 ≥5 万局;
- 训练:机器 A 上训 belief 网络与模仿策略(1050 Ti 可用则用,否则 CPU——模型是小 MLP,均可行);训练时机器 B 可继续为 S5 屯数据;
- 验收沿 S4 spec 不变(belief 优于先验/校准/尾盘最大提升;策略零非法动作等)。

**S5(预计:引擎提速后,单轮完整训练 1~3 个空调夜)**:
- 机器 A:rollout(留 1~2 线程给 learner 更新,其余采样)+ PPO 训练;
- 机器 B:每个里程碑 checkpoint 拷过去跑竞技场评估(1 万局对 S3 双赛道),**评估与训练并行**,机器 A 训练不中断;
- 评估结果回传格式沿 S4 arena 报告。

**验收**:S4/S5 各自 spec 的验收标准原样适用;本任务只验收编排本身——双机产物正确汇合、评估不阻塞训练、全程无 OOM 无过热停机。

### 任务 5:吞吐台账与"转云"判据

**做什么**:每个阶段结束记录实际吞吐与耗时,维护一张台账(markdown 表),并按下面判据决定是否转云:

- **转云判据(满足任一)**:
  1. 任务 1 提速后,S5 单轮完整训练(含课程全程)仍 >7 个空调夜;
  2. 迭代密集期需要同时跑 ≥3 组对比实验(本地只有一台训练机,无弹性);
  3. 机器 A 出现硬件不稳(过热/内存报错),修复成本高于云费;
- 转云方式:租 32 核 CPU 实例(约 ¥18/天,按量冒烟半天后再包日),代码与数据经任务 2 的压缩分片直接迁移;本地两台降级为开发/数据/评估角色;
- 云上注意:Linux 环境已由机器 B 验证过,无额外适配成本。

**验收**:台账存在且随阶段更新;转云判据写入 README 显眼处。

## 四、全局技术约束

- 一切以 S4/S5 spec 的训练与验收要求为准,本文档只解决"在哪跑、怎么跑得起";两者冲突时以 S4/S5 spec 为准并回报人类;
- **行为等价红线**:任何性能优化(Cython、并行化)不得改变引擎/编码/训练逻辑的输出——同种子对拍是每次优化的必过关;
- 数据文件一律 gzip;所有长任务必须断点友好;所有脚本两台机器可运行(Windows + Ubuntu,路径用相对路径/pathlib);
- 机器 A 满载仅限空调时段 + 温度保险常开;机器 B 可常驻;
- 预算纪律:本方案硬件投入上限 ¥600(内存+硅脂+可选硬盘);超出即触发"转云判据"重新算账。

## 五、执行顺序

任务 0(硬件/环境,人机配合)→ 任务 1(Cython 提速,关键路径)→ 任务 2(双机数据工厂)→ S4 执行(按其 spec)→ 任务 3(暂停/恢复纪律,S5 开始前就位)→ S5 执行(任务 4 编排)→ 全程维护任务 5 台账。

## 六、我(人类)负责

- 完成任务 0 的采购与动手项(内存、清灰、磁盘清理);
- 提供每晚可训练的时间窗(告知脚本调度配置);
- 阶段性查看任务 5 台账,在触发转云判据时拍板;
- S4/S5 的人工验收项(belief 直觉抽查、高手行为清单评审)照旧。

## 七、本方案成功标志

**引擎提速 ≥10 倍且行为逐分等价;双机并行数据工厂稳定产出;训练可随时中断恢复且曲线衔接;S4 在本地两晚内完成并通过其全部验收;S5 单轮完整训练在 ≤7 个空调夜内完成并通过双赛道验收;全程硬件投入 ≤¥600、无过热事故。**

达成后,"持续迭代"的成本结构变为:每轮实验 ≈ 几个空调夜的电费,试错自由度大幅提升;云仅作为弹性溢出通道。
